# Phase 2 — UI Design Contract

> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Tailwind CSS v4 primitives only — inherits Phase 1 decision) |
| Preset | not applicable |
| Component library | none — hand-built with Tailwind |
| Icon library | Lucide React (`lucide-react`) — inherits Phase 1 decision |

## Spacing Scale

Inherits Phase 1 scale. No changes.

| Token | Tailwind | px | Usage |
|-------|----------|----|-------|
| xs | `gap-1` / `p-1` | 4px | Icon-to-label gap, allergen checkbox gap |
| sm | `gap-2` / `p-2` | 8px | Table cell inner padding, badge padding |
| md | `gap-4` / `p-4` | 16px | Form field vertical gap, modal section gap |
| lg | `gap-6` / `p-6` | 24px | Modal padding, card padding |
| xl | `gap-8` / `p-8` | 32px | Page content padding (inherited from layout) |
| 2xl | `gap-12` / `p-12` | 48px | Not used in Phase 2 |
| 3xl | `gap-16` / `p-16` | 64px | Not used in Phase 2 |

**Exceptions:**
- Touch targets on all interactive elements: minimum `min-h-[44px] min-w-[44px]` (WCAG 2.5.5 AA)
- Order buttons (↑↓): `min-h-[32px] min-w-[32px]` — tight row density justified; these are secondary controls
- Allergen checkbox cells: `min-h-[36px]` — checkbox + label pair, not a standalone touch target
- Modal overlay: covers full viewport with `inset-0`

## 1. Layout & Structure

### Page Layout — Categories (`/dashboard/categories`) and Dishes (`/dashboard/dishes`)

Both pages slot into the existing admin shell from `app/(admin)/layout.tsx`. The shell provides:
- Fixed left sidebar (`w-64`, `bg-white`, `border-r border-brand-acento`)
- Top header bar (`h-14`, `bg-white`, `border-b border-brand-acento`) — page title updates per route
- `<main class="flex-1 overflow-y-auto p-8 flex flex-col">`

**Page title in header bar:**
- Categories route → `Categorías`
- Dishes route → `Platos`

The header title is currently hardcoded as `"Dashboard"` in `app/(admin)/layout.tsx`. Phase 2 must make it dynamic. Strategy: read route from `usePathname()` in a client wrapper, or pass a `title` prop. Preferred: extract header into `DashboardHeader` client component that reads `usePathname()`.

```
<div class="flex flex-col gap-6 flex-1">

  [Page header row]
  <div class="flex items-center justify-between">

  [Content area]
  <div class="bg-white rounded-lg shadow-sm border border-brand-acento overflow-hidden">
    {table or list — see component contracts below}
```

**Sidebar nav items to enable:** Set `enabled: true` for `Categorías` (`/dashboard/categories`) and `Platos` (`/dashboard/dishes`) in `components/dashboard/Sidebar.tsx`.

**Backdrop:** `<dialog>` native `::backdrop` is not used. The dialog is rendered over a custom overlay div:

## 3. Interaction & States

### Loading States

**Page-level (initial data load):** Server Components render the page with data server-side. No skeleton UI needed — the page is only rendered once data is available.

**Button loading (during Server Action):**
- Disable button + add `disabled:opacity-50 disabled:cursor-not-allowed`
- Replace button text with: `<Loader2 size={14} class="animate-spin inline mr-2" />{action verb}...`
- Examples: `Guardando...`, `Eliminando...`, `Subiendo...`

**Availability toggle optimistic update:**
- Toggle state updates immediately in local state
- Server Action fires in background
- On error: revert state + show error toast

**Image upload progress:**
- After file selection, show inline `Subiendo imagen...` with spinner below the file input
- On success: show thumbnail preview
- On error: show inline error message below file input (see Copywriting)

### Input States

| State | Border | Ring | Background |
|-------|--------|------|------------|
| Default | `border-gray-200` | none | `bg-white` |
| Focus | `border-brand-principal` | `ring-1 ring-brand-principal` | `bg-white` |
| Error | `border-red-500` | `ring-1 ring-red-500` | `bg-white` |
| Disabled | `border-gray-100` | none | `bg-gray-50` |

### Form Validation States

- Client-side: validate on `blur` for individual fields; validate all on submit
- Show error message below the field (`text-xs text-red-600` with `role="alert"`)
- Required field marker: `*` in `text-red-500` next to label
- Submit button stays enabled until the user attempts to submit; then disable during loading

### Delete Confirmation State Machine

1. Idle → user clicks delete icon button
2. Inline confirmation prompt appears (action buttons hidden)
3a. User clicks "Sí, eliminar" → action buttons disabled + `Eliminando...` state → success/error toast
3b. User clicks "Cancelar" → returns to idle (action buttons reappear)
